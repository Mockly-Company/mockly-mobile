# PR 메시지 템플릿

이 문서는 PR 메시지의 표준 템플릿과 각 섹션별 작성 가이드를 제공합니다.

## 기본 템플릿

```markdown
## 요약

[2-3문장으로 전체 변경사항의 핵심 요약]

## 주요 변경사항

- [가장 중요한 변경사항 1]
- [가장 중요한 변경사항 2]
- [가장 중요한 변경사항 3]
- ...

## 변경 내용

### [카테고리 1]

- [세부 변경사항 1]
- [세부 변경사항 2]

### [카테고리 2]

- [세부 변경사항 1]
- [세부 변경사항 2]

### [카테고리 3]

...

## 테스트

### 추가된 테스트

- [ ] [테스트 항목 1]
- [ ] [테스트 항목 2]

### 수동 테스트 가이드

1. [테스트 단계 1]
2. [테스트 단계 2]
3. [예상 결과]

### 테스트 환경

- [환경 정보 1]
- [환경 정보 2]

## 스크린샷

<!-- 여기에 스크린샷을 추가해주세요 -->

## 관련 이슈

- #[이슈번호] - [이슈 제목]

## 완료

- [x] [완료된 작업 1]
- [x] [완료된 작업 2]
- [x] [완료된 작업 3]
```

## 섹션별 상세 가이드

### 1. 요약 (Summary)

**목적**: PR의 핵심 내용을 빠르게 파악할 수 있도록 합니다.

**작성 방법:**

- 2-3문장으로 간결하게 작성
- 왜 필요한지 (배경/문제)
- 무엇을 했는지 (해결책)
- 어떤 결과인지 (효과)

**좋은 예시:**

```
React Native 프로젝트에 일관된 UI/UX를 제공하고 개발 생산성을 향상시키기 위한
디자인 시스템을 구축했습니다. NativeWind에서 twrnc로 마이그레이션하고,
재사용 가능한 UI 컴포넌트와 레이아웃 컴포넌트를 구현했습니다.
```

**나쁜 예시:**

```
컴포넌트 추가했습니다.
```

### 2. 주요 변경사항 (Key Changes)

**목적**: 가장 중요한 변경사항을 한눈에 파악할 수 있도록 합니다.

**작성 방법:**

- 3-7개의 bullet point
- 우선순위 순으로 나열
- 각 항목은 한 문장으로 간결하게

**예시:**

```markdown
## 주요 변경사항

- 스타일링 라이브러리 전환: NativeWind → twrnc로 마이그레이션하여 RN 0.82 호환성 개선
- 디자인 토큰 정의: Tailwind config에 색상, 타이포그래피, spacing 토큰 적용
- 기본 UI 컴포넌트: Button, Input, Text, Card 컴포넌트 구현
- 레이아웃 컴포넌트: Stack, Grid, Spacer 컴포넌트 추가
- Storybook 설정: RN 네이티브 환경과 Web 환경 모두 지원하는 Storybook 구축
```

### 3. 변경 내용 (Detailed Changes)

**목적**: 카테고리별로 세부 변경사항을 상세히 설명합니다.

**작성 방법:**

- 적절한 카테고리로 그룹화
- 각 카테고리 내에서 관련 변경사항을 나열
- 코드 블록이나 예시를 포함할 수 있음

**프론트엔드 예시:**

```markdown
## 변경 내용

### 빌드 및 설정

- twrnc 라이브러리로 마이그레이션 완료
- @gorhom/bottom-sheet RN 0.82 호환성 패치 적용
- Storybook RN + Web 빌드 환경 구성
- TypeScript 타입 정의 및 설정 개선

### 디자인 토큰

colors: {
primary, secondary, accent, background, surface,
text, textSecondary, border, error, warning, success
}
spacing: { xs, sm, md, lg, xl, 2xl }
fontSize: { xs, sm, md, lg, xl, 2xl }

### 구현된 컴포넌트

#### UI 컴포넌트

**Button**

- variant: primary, secondary, outline
- size: small, medium, large
- props 기반 커스터마이징 지원

**Input**

- label, error props 지원
- 포커스 상태 스타일링

**Text**

- variant: h1, h2, h3, body, caption
- color, weight props 지원

**Card**

- variant: elevated, outlined, filled
- padding 옵션

#### 레이아웃 컴포넌트

- **Stack**: 수직/수평 정렬, spacing 조절
- **Grid**: 그리드 레이아웃
- **Spacer**: 간격 조정용 컴포넌트

### 문서화

- 각 컴포넌트별 Storybook 스토리 작성 완료
- Props 예시 및 사용법 문서화
```

**백엔드 예시:**

```markdown
## 변경 내용

### API 엔드포인트

- POST /api/auth/login - 사용자 로그인
- POST /api/auth/register - 회원가입
- GET /api/users/me - 현재 사용자 정보 조회

### 비즈니스 로직

- JWT 토큰 기반 인증 구현
- bcrypt를 사용한 비밀번호 해싱
- 이메일 중복 검증 로직

### 데이터베이스

- User 모델 생성 (email, password, name, createdAt)
- 마이그레이션 파일 추가

### 미들웨어

- authMiddleware: JWT 검증 미들웨어 구현
- errorHandler: 통합 에러 핸들링 미들웨어
```

### 4. 테스트 (Testing)

**목적**: 테스트 관련 모든 정보를 제공하여 리뷰어가 검증 방법을 이해할 수 있도록 합니다.

**작성 방법:**

- 추가된 자동화 테스트
- 수동 테스트 방법
- 테스트 환경 및 주의사항

**예시:**

```markdown
## 테스트

### 추가된 테스트

- [x] 단위 테스트: Button 컴포넌트 렌더링 테스트
- [x] 단위 테스트: useAuth 훅 로직 테스트
- [x] 통합 테스트: 로그인 플로우 E2E 테스트
- [x] Storybook: 모든 컴포넌트 상태별 스토리 추가

### 수동 테스트 가이드

1. Storybook 실행: `npm run storybook`
2. 각 컴포넌트의 다양한 variant/size/state 확인
3. 로그인 화면에서 실제 로그인 플로우 테스트
4. 에러 케이스 확인 (잘못된 이메일/비밀번호 입력)

### 테스트 환경

- iOS 시뮬레이터 (iOS 16.0) 동작 확인
- Android 에뮬레이터 (API 33) 동작 확인
- Storybook Web 환경에서 모든 컴포넌트 렌더링 확인
- Jest 테스트 100% 통과

### 알려진 이슈

- Android에서 Input 컴포넌트의 포커스 애니메이션이 iOS보다 느림 (추후 최적화 예정)
```

### 5. 스크린샷 (Screenshots)

**목적**: 시각적 변경사항을 보여줍니다.

**작성 방법:**

- UI 변경이 있는 경우 반드시 포함
- Before/After 비교가 유용한 경우 양쪽 모두 포함
- 각 스크린샷에 설명 추가

**예시:**

```markdown
## 스크린샷

### 버튼 컴포넌트

![Button variants](이미지_URL)

### 카드 컴포넌트

![Card examples](이미지_URL)

### 로그인 화면

![Login screen](이미지_URL)

### Before / After

| Before             | After              |
| ------------------ | ------------------ |
| ![old](이미지_URL) | ![new](이미지_URL) |
```

### 6. 관련 이슈 (Related Issues)

**목적**: 이슈 트래커와 연결하여 작업 컨텍스트를 제공합니다.

**작성 방법:**

- 관련 이슈 번호와 간단한 설명
- 적절한 키워드 사용 (Closes, Fixes, Relates to)

**키워드 의미:**

- `Closes #번호`: PR 머지 시 이슈 자동 종료
- `Fixes #번호`: 버그 수정 시 사용 (머지 시 이슈 종료)
- `Resolves #번호`: 이슈 해결 (머지 시 이슈 종료)
- `Relates to #번호`: 관련 있지만 종료하지 않음

**예시:**

```markdown
## 관련 이슈

- Closes #123 - 디자인 시스템 구축
- Fixes #456 - Button 컴포넌트 스타일 버그
- Relates to #789 - 전체 UI 리뉴얼 프로젝트
```

### 7. 완료 (Completed Tasks)

**목적**: 구현된 모든 작업을 체크리스트로 정리합니다.

**작성 방법:**

- 실제로 완료된 작업만 체크
- 주요 작업을 누락 없이 포함
- 논리적 순서로 나열

**예시:**

```markdown
## 완료

- [x] 디자인 토큰(색상, 폰트, spacing) Tailwind config 적용
- [x] 기본 UI 컴포넌트 4개 구현 (Button, Input, Text, Card)
- [x] 레이아웃 컴포넌트 3개 구현 (Stack, Grid, Spacer)
- [x] TypeScript 타입 정의 포함
- [x] Storybook 문서화 완료
- [x] 단위 테스트 작성
- [x] README 사용 가이드 작성
```

## 템플릿 변형

프로젝트나 상황에 따라 템플릿을 유연하게 조정할 수 있습니다:

### 버그 수정용 템플릿

```markdown
## 문제 설명

[버그에 대한 설명]

## 원인

[버그의 원인 분석]

## 해결 방법

[어떻게 수정했는지]

## 테스트

[재현 방법 및 검증]

## 관련 이슈

- Fixes #버그*이슈*번호
```

### 리팩터링용 템플릿

```markdown
## 리팩터링 목적

[왜 리팩터링이 필요한지]

## 주요 변경사항

[어떤 구조적 변경을 했는지]

## 변경 전/후 비교

[코드 예시 또는 구조도]

## 테스트

[기능이 동일하게 동작하는지 확인]

## 완료

[리팩터링된 항목 체크리스트]
```

### 성능 개선용 템플릿

```markdown
## 성능 문제

[어떤 성능 문제가 있었는지]

## 개선 방법

[어떻게 최적화했는지]

## 성능 측정 결과

[Before/After 성능 비교]

## 테스트

[성능 개선 검증 방법]
```

## 작성 팁

1. **명확한 언어 사용**: 모호한 표현 대신 구체적인 용어 사용
2. **적절한 분량**: 너무 짧으면 정보 부족, 너무 길면 읽기 어려움
3. **리뷰어 관점**: 리뷰어가 알아야 할 정보 우선 제공
4. **일관성 유지**: 같은 프로젝트 내에서 비슷한 형식 사용
5. **시각적 구조**: 섹션, 리스트, 코드 블록으로 가독성 향상
